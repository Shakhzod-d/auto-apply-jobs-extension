# Auto Apply Jobs — Chrome Extension

Manifest V3 extension that autofills (and optionally auto-submits) LinkedIn
job applications — both Easy Apply and jobs "managed off LinkedIn" that
redirect to the employer's own application page — syncing every application
and any unanswered screening questions to the [dashboard](../dashboard) app.

It runs entirely inside your own already-logged-in browser tab — it never
stores or handles your LinkedIn credentials.

## Development

```bash
npm install
npm run dev
```

Then in Chrome: `chrome://extensions` → enable Developer mode → **Load
unpacked** → select this project's `dist/` folder (run `npm run build` first,
or `npm run dev` and reload after each change — crxjs HMR handles most
updates automatically).

## Structure

- `manifest.config.ts` — MV3 manifest (generated by `@crxjs/vite-plugin`).
- `src/background/` — service worker; syncs Profile/QuestionBank/Settings
  from the dashboard, opens/navigates the tab a bulk run works in, and
  detects + arms the external-apply tab a LinkedIn "Apply" click opens.
- `src/content/linkedin.ts` — two things: (1) autofilling any Easy Apply
  modal that opens, whether a human or the bulk driver opened it, and (2)
  the bulk-apply driver loop, which works through the current search
  results page clicking Apply (Easy Apply or external) on each
  not-yet-applied listing.
- `src/content/external-apply.ts` — handles whatever page a "managed off
  LinkedIn" job's Apply button opens (Greenhouse, Lever, Workday, or a
  fully custom company site). Present on every page via an `<all_urls>`
  content script match, but completely passive until the background sends
  it a targeted message for that one specific tab — never runs on a page
  you're just browsing.
- `src/popup/` — toolbar popup: submit-mode toggle, start/stop the bulk
  run, live progress (processed/submitted/pending/blocked/failed) and
  today's cap usage.
- `src/options/` — settings page: dashboard URL + API token.

## How bulk apply works

1. Set **job search keywords** on the dashboard's Settings page.
2. Click **Start applying on LinkedIn** in the popup (this also prompts for
   all-sites access, needed for external-apply.ts — see below).
3. The content script works through the results list: click a card, then
   branch on what its Apply button actually is:
   - **Easy Apply**: fill the modal's steps, pace, next card.
   - **External** ("managed off LinkedIn"): click Apply, which opens the
     employer's page in a new tab. The background worker arms
     `external-apply.ts` on it once loaded; that script fills whatever
     fields it can generically match, then **always stops for your own
     review and submit click** — it never auto-submits on a site it
     doesn't recognize, unlike LinkedIn's own consistent Easy Apply UI.
     The main loop does **not** wait for you to review that tab — it
     paces and moves straight to the next job, so external-apply tabs
     accumulate for you to work through at your own pace without slowing
     down the run.
4. If a required question has no matching answer, that application is
   **not** blocked on indefinitely — the question is reported to the
   dashboard, the draft is saved (LinkedIn's own "save for later" for Easy
   Apply; left filled-in for external), and the loop moves on immediately.
   Answer it from the dashboard's Needs-your-input queue whenever
   convenient.
5. The loop stops on: daily cap reached (overall or per-platform),
   LinkedIn disabled in Settings, no more results, an Easy Apply
   filled-and-waiting-for-review (since that happens in *this* tab and
   would otherwise barrel past you), or you clicking **Stop**.

## Permissions

- `https://www.linkedin.com/*` — required, declared up front.
- The dashboard's own origin — requested via `chrome.permissions.request()`
  when you save it in Options.
- `*://*/*` (all sites) — requested via `chrome.permissions.request()` when
  you click **Start applying**, since external-apply.ts needs to be allowed
  to run on whatever ATS domain a job happens to redirect to. It's a broad
  grant; the content script itself stays inert everywhere except the one
  tab the background explicitly arms.

## Safety defaults

Submit mode defaults to **review** (fill the form, stop before submitting)
and pacing/daily caps are enforced from the dashboard's Settings — see the
root plan doc for why. Raise the daily cap and switch to auto-submit only
after you've verified field-matching accuracy on a handful of real
applications. Note that auto-submit only ever applies to Easy Apply —
external-apply always requires your own review and submit click.
